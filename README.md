# 一般社団法人 Grow福祉協会 公式サイト

**「一人ひとりの成長（Grow）を支え、地域とともに歩む。」**

このリポジトリは、一般社団法人Grow福祉協会（英語表記：Grow Welfare Association）の公式Webサイトのソースコード一式です。HTML5・CSS3・JavaScript（Vanilla JS）のみで構成されており、外部ライブラリやフレームワークに依存せず、ダウンロードしてブラウザで開くだけで完全に動作する静的サイトです。Firebase Hostingなどの静的ホスティングサービスにそのままデプロイできます。

このREADMEでは、サイトの目的やコンセプトから、ファイル構成、各ページの内容、デザインの考え方、実装されている機能、ローカルでの確認方法、Firebase Hostingへのデプロイ手順、今後の運用でファイルを差し替える際の注意点、アクセシビリティやSEOに関する配慮、既知の制約や今後の改善候補まで、できる限り詳しく記載しています。制作担当者が変わった場合や、外部の制作会社に運用を引き継ぐ場合にも、このドキュメントだけで全体像を把握できることを目指しています。

## 1. プロジェクトの概要

本プロジェクトは、一般社団法人Grow福祉協会のブランディングと情報発信を目的とした公式コーポレートサイトです。法人の理念、代表理事のメッセージ、現在運営中の事業（放課後等デイサービス「みらい」）、そして将来的に展開を目指す多数の事業構想、採用情報、お問い合わせ導線をひとつのサイトにまとめています。

制作にあたっては、以下の条件を満たすことを前提としています。

- HTML5・CSS3・JavaScript（Vanilla JS）のみで構築し、React・Vue・jQueryなどの外部ライブラリやCDN経由のフォント・アイコンフォントなどは一切使用しないこと。
- PC・タブレット・スマートフォンのいずれの画面幅でも快適に閲覧できるレスポンシブデザインであること。
- スクロールに連動したフェードイン・スライドインアニメーションを備え、モダンで温かみのある「福祉法人らしい」印象のデザインであること。
- 白・ブルー・グリーンを基調とした配色で、法人の理念である「安心」「信頼」「成長」「地域とのつながり」を視覚的に表現すること。
- アクセシビリティ（スクリーンリーダー対応、キーボード操作対応、モーション低減設定への配慮など）を考慮すること。
- SEO対策として適切なmetaタグ・OGPタグ・faviconを設定すること。
- ハンバーガーメニュー、スムーズスクロール、スクロール連動アニメーション、フォームバリデーションなどをJavaScriptで実装すること。
- ヘッダー・フッターは全ページ共通のデザイン・構成とすること。
- 開設前・構想段階の事業については、実際に運営中であるかのような誤解を利用者に与えないよう、「構想中」「開設準備中」といった表記を明確に行うこと。

これらの要件を踏まえ、5つのHTMLページ（トップページ／法人案内／事業紹介／採用情報／お問い合わせ）と、共通のスタイルシート・共通のJavaScriptファイル、およびSVG形式の画像アセット（ロゴ・favicon・OGP画像・ヒーロー背景・代表理事プロフィール写真のプレースホルダー）から構成されるサイトを制作しました。

## 2. 法人紹介とサイトのコンセプト

### 2.1 法人名

- 正式名称（日本語）：一般社団法人 Grow福祉協会
- 英語表記：Grow Welfare Association

### 2.2 キャッチコピー

「一人ひとりの成長（Grow）を支え、地域とともに歩む。」

このキャッチコピーは、サイト全体を貫くメッセージであり、トップページのヒーローセクションの見出しをはじめ、法人案内ページの理念セクション、フッターのブランド説明文など、複数箇所で繰り返し用いています。単なるスローガンとしてではなく、サイトを訪れた誰もが最初にこの言葉に触れ、法人の姿勢を理解できるように設計しています。

### 2.3 デザインテーマ

デザインのテーマは「優しさ・信頼・未来・地域」の4つのキーワードに集約されます。これを視覚的に表現するために、以下のようなデザイン上の工夫を行いました。

- **優しさ**：角丸を多用したカードデザイン、やわらかいグラデーション、余白を十分に取ったレイアウトによって、威圧感のない柔和な印象を作っています。
- **信頼**：ネイビーを見出しの基本色に採用し、情報の階層を明確にすることで、誠実で落ち着いた印象を持たせています。
- **未来**：ブルーからグリーンへと変化するグラデーション（サイト全体で繰り返し使用している「ブランドグラデーション」）は、空（未来・可能性）から若葉（成長）へと視線が導かれるようなイメージを意図しています。
- **地域**：法人案内ページのアクセス情報や、事業紹介ページにおける地域密着型の事業紹介を通じて、地域との関わりを随所で強調しています。

### 2.4 サイトのシグネチャー要素：「芽」のモチーフ

法人名である「Grow（成長）」を象徴するビジュアルとして、サイト全体を通じて「芽が伸びていく」形状のラインアートを繰り返し使用しています。具体的には以下の場所に登場します。

- ヘッダーおよびフッターのロゴマーク（円形のエンブレムの中に、茎から左右に葉が伸びるシルエット）
- favicon（同じロゴマークを簡略化したもの）
- トップページのヒーローセクション右側に配置した、SVGアニメーションによる「育つイラスト」（ページ読み込み時に茎と葉が描かれていくアニメーション）
- 法人理念セクション、採用情報の「求める人物像」セクション、お問い合わせページの連絡先アイコンなど、随所に配置した「sprout-icon（芽アイコン）」

この一貫したモチーフにより、テキストで理念を説明するだけでなく、視覚的にも「成長」というコンセプトが伝わるようにしています。

## 3. ファイル構成

リポジトリ直下（またはZIP解凍後）のファイル構成は以下の通りです。

```
grow-fukushi/
├── index.html              … トップページ
├── about.html               … 法人案内ページ
├── services.html            … 事業紹介ページ
├── recruit.html              … 採用情報ページ
├── contact.html              … お問い合わせページ
├── style.css                 … 全ページ共通のスタイルシート
├── script.js                  … 全ページ共通のJavaScriptファイル
├── favicon.svg                … favicon（サイトアイコン）
├── README.md                  … 本ドキュメント
└── images/
    ├── hero-bg.svg              … トップページのヒーロー背景（抽象的な丘と空のイラスト）
```

すべてのHTMLファイルは同一階層に配置されており、相対パス（`./style.css`、`./script.js`、`./images/〇〇.svg`など）でCSS・JS・画像を読み込んでいます。そのため、フォルダ構成を変更しない限り、どの環境にアップロードしてもリンク切れは発生しません。フォルダ名を変更したり、サブディレクトリに配置したりする場合は、各HTMLファイル内の相対パスとの整合性にご注意ください。

6. **事業紹介（抜粋）**：現在の事業と代表的な構想中の事業を3枚のカードで紹介し、事業紹介ページへの導線を設置

### 4.2 法人案内（about.html）

法人の詳細情報をまとめたページです。

### 4.5 お問い合わせ（contact.html）

問い合わせフォームを中心としたページです。

- 左側に法人の連絡先情報（メール・電話・所在地）をまとめたカードを固定表示（スクロール追従、ただしタブレット・スマートフォン幅では通常表示に切り替え）
- 右側に入力フォームを配置し、氏名・フリガナ・メールアドレス・電話番号（任意）・お問い合わせ種別（プルダウン）・お問い合わせ内容（テキストエリア）・個人情報保護方針への同意チェックボックスを備えています。

## 5. デザインコンセプトと配色・タイポグラフィ

### 5.1 カラーパレット

`style.css`の冒頭にCSSカスタムプロパティ（変数）としてすべての色を定義しています。主な色は以下の通りです。

| 変数名 | 色 | 用途 |
|---|---|---|
| `--color-white` | #ffffff | 基本背景色 |
| `--color-mist` | #f4f9fb | セクション背景の淡いブルー |
| `--color-sky` | #4e93c4 | メインブルー（ボタン・アクセント） |
| `--color-sky-deep` | #2c6b99 | ブルーのホバー・強調色 |
| `--color-leaf` | #5cb894 | メイングリーン（ボタン・アクセント） |
| `--color-leaf-deep` | #3e9973 | グリーンのホバー・強調色 |
| `--color-navy` | #1f3a52 | 見出し・フッター背景色 |
| `--color-ink` | #45535e | 本文テキスト色 |
| `--color-warn` | #d97a5f | 「構想中」バッジなどの注意喚起色 |

これらの変数を使うことで、将来的にブランドカラーを変更したい場合でも、`style.css`冒頭の`:root`ブロックの値を書き換えるだけでサイト全体の配色を一括変更できます。

### 5.2 グラデーション

サイトのシンボルとして、ブルーからグリーンへ向かう斜めのグラデーション（`--gradient-brand`）を、主要なボタン、CTAセクションの背景、ロゴのアクセントカラーなどに繰り返し使用しています。これは「空（未来）から若葉（成長）へ」というイメージを表現したものです。

### 5.3 タイポグラフィ

外部フォント（Google FontsなどのCDN経由のWebフォント）を一切使用しない方針としたため、OSに標準搭載されている日本語フォントをシステムフォントスタックとして指定しています。

```css
--font-base: "Hiragino Sans", "Hiragino Kaku Gothic ProN", "Yu Gothic Medium",
  "Yu Gothic", "Meiryo", -apple-system, BlinkMacSystemFont, sans-serif;
```

これにより、Macでは「ヒラギノ角ゴ」、Windowsでは「游ゴシック」または「メイリオ」が自動的に適用され、追加の通信を発生させることなく、どの環境でも読みやすい日本語表示が可能です。見出しには本文よりも太いウェイト（`font-weight: 700`）を使用し、階層構造を明確にしています。

### 5.4 カードデザインとガラス風（グラスモーフィズム）デザイン

情報を提示する多くの箇所で、半透明の背景に`backdrop-filter: blur()`を適用した「グラスモーフィズム」スタイルのカード（`.card`クラス）を採用しています。角丸（`border-radius`）を大きめに設定し、柔らかい影（`box-shadow`）を添えることで、福祉法人らしい親しみやすさを演出しています。ヘッダーにも同様の半透明・ぼかし効果を適用し、スクロールしても背景のコンテンツがうっすら透けて見えるようにしています。

## 6. 実装されている主な機能

すべての機能は`script.js`内に日本語コメント付きで実装されており、それぞれ独立した関数として分割されています。`DOMContentLoaded`イベント発火時に、以下の9つの初期化関数がまとめて呼び出されます。

1. **`initHeaderScroll()`**：スクロール量に応じてヘッダーに`is-scrolled`クラスを付与し、ヘッダーの高さと影を変化させます。
2. **`initHamburgerMenu()`**：モバイル幅でのナビゲーション開閉を制御します。`aria-expanded`属性の切り替え、背景オーバーレイの表示、ESCキーでの閉じる操作、画面リサイズ時の状態リセットなど、アクセシビリティに配慮した実装です。
3. **`initSmoothScroll()`**：ページ内アンカーリンク（`#`を含むリンク）をクリックした際に、ヘッダーの高さ分をオフセットしながら滑らかにスクロールします。
4. **`initScrollReveal()`**：`IntersectionObserver`を用いて、要素が画面内に入ったタイミングで`.is-visible`クラスを付与し、CSS側のトランジションでフェードイン・スライドインを発火させます。`.reveal`（下からフェードイン）、`.reveal-left`（左からスライドイン）、`.reveal-right`（右からスライドイン）、`.reveal-stagger`（子要素を時間差で順番にフェードイン）の4種類のアニメーションパターンを用意しています。
5. **`initToTopButton()`**：一定量スクロールすると画面右下にページトップボタンを表示し、クリックでページ最上部へスムーズスクロールします。
6. **`initNewsSlider()`**：トップページのお知らせセクションをカルーセル化し、5秒ごとに自動でスライドを切り替えます。ドットナビゲーションのクリックによる手動切り替えにも対応し、マウスホバー中は自動切り替えを一時停止します。
7. **`initServiceFilter()`**：事業紹介ページのタブ（すべて／開設中／構想中）をクリックすると、`data-status`属性に基づいて該当するカードのみを表示します。
8. **`initFaqAccordion()`**：採用情報ページのFAQ項目をクリックすると、回答部分の開閉をアニメーション付きで切り替えます。
9. **`initContactForm()`**：お問い合わせフォームの入力チェックを行います。氏名・フリガナ・お問い合わせ種別・お問い合わせ内容は空欄でないことを確認し、メールアドレスは正規表現によって形式をチェックします。電話番号は任意項目ですが、入力された場合のみ形式チェックを行います。エラーがある場合は該当項目に赤枠とエラーメッセージを表示し、すべて正しい場合は送信完了メッセージを表示してフォームをリセットします。

また、`prefers-reduced-motion: reduce`が設定されている環境（OSやブラウザで「視差効果を減らす」等の設定を有効にしているユーザー）に対しては、CSS側でアニメーションやスムーズスクロールを実質的に無効化し、視覚的な動きに敏感なユーザーへの配慮も行っています。

## 7. コードの構造とメンテナンス性への配慮

### 7.1 CSSの構造

`style.css`は以下の7つのセクションに分割されており、ファイル冒頭にも目次コメントを記載しています。

1. デザイントークン（変数定義）
2. リセット・基本設定
3. 共通レイアウト（ヘッダー／フッター／コンテナ）
4. コンポーネント（ボタン・カード・フォーム等）
5. ページ別スタイル
6. アニメーション
7. レスポンシブ対応

新しいセクションやコンポーネントを追加する際は、対応するセクションに追記することで、ファイル全体の見通しを保てるようにしています。また、クラス名は`BEM`ほど厳密ではないものの、役割が推測しやすい命名（例：`.hero-copy`、`.hero-visual`、`.service-card`、`.faq-question`など）を心がけています。

### 7.2 JavaScriptの構造

`script.js`は機能ごとに関数を分割しており、1つの関数が1つの役割のみを持つように設計しています。新しい機能を追加する場合は、新しい`init〇〇()`関数を作成し、ファイル末尾付近の`DOMContentLoaded`イベントリスナー内に呼び出しを追加するだけで拡張できます。

### 7.3 HTMLの構造

各ページのヘッダー・フッターのHTMLは完全に共通の構造となっており、コピー＆ペーストで管理しています（サーバーサイドインクルードやテンプレートエンジンを使用していないため、静的サイトとしてどの環境でも動作することを優先した結果です）。ナビゲーション内の該当ページのリンクには`aria-current="page"`属性を付与し、現在地を示すハイライト表示を行っています。ヘッダー・フッターのデザインや文言を変更する場合は、5つのHTMLファイルすべてに同じ修正を適用する必要がある点にご注意ください。

## 8. アクセシビリティへの配慮

本サイトでは、以下のようなアクセシビリティ対応を行っています。

- **スキップリンク**：各ページの先頭に「メインコンテンツへスキップ」というリンクを設置し、キーボード操作時に繰り返しのナビゲーションを読み飛ばしてメインコンテンツへ直接移動できるようにしています。
- **フォーカスの可視化**：`:focus-visible`セレクタを用いて、キーボード操作時にはっきりとしたフォーカスリングが表示されるようにしています。マウス操作時には過剰な視覚的ノイズを避けるため、`:focus-visible`を採用しています。
- **`aria-expanded`・`aria-controls`属性**：ハンバーガーメニューやFAQのアコーディオンボタンには、開閉状態を示す`aria-expanded`属性を付与し、スクリーンリーダー利用者にも状態変化が伝わるようにしています。
- **`aria-label`**：アイコンのみのボタン（ハンバーガーメニュー、ページトップボタンなど）には、視覚的なラベルがない代わりに`aria-label`で用途を説明しています。
- **`alt`属性**：画像には内容を説明する`alt`属性を設定しています。プレースホルダー画像についても「（準備中）」である旨をテキストで補足しています。
- **モーション低減設定への対応**：`prefers-reduced-motion: reduce`が有効な環境では、アニメーションの時間を実質的にゼロに近づけ、スムーズスクロールも即時移動に切り替わるようにしています。
- **フォームのエラー通知**：フォームのエラーメッセージには`role="alert"`を付与し、送信完了・エラーメッセージ全体には`role="status"`と`aria-live="polite"`を付与することで、スクリーンリーダー利用者にも動的な変化が読み上げられるようにしています。

これらはあくまで基本的な対応であり、JIS X 8341-3やWCAG 2.1のAAレベルへの準拠を正式に検証したものではありません。本格的な公共性の高いサイトとして運用する場合は、専門のアクセシビリティ診断を受けることを推奨します。

## 9. SEO・OGP・favicon設定について

各HTMLファイルの`<head>`内には、以下の要素を設定しています。

- `<title>`：ページごとに固有のタイトルを設定（例：「法人案内｜一般社団法人 Grow福祉協会」）
- `<meta name="description">`：ページごとの要約文
- `<meta name="keywords">`：関連キーワード
- `<meta name="author">`：法人名
- `<link rel="canonical">`：正規URLの指定（相対パスで記載しているため、実際の公開ドメインが確定した際には絶対URLへの変更を推奨します）
- OGPタグ（`og:type`、`og:title`、`og:description`、`og:image`、`og:url`、`og:site_name`、`og:locale`）：Facebook・LINE・Slackなどでシェアされた際に表示される情報を制御します
- `<meta name="twitter:card" content="summary_large_image">`：Xでシェアされた際の表示形式を指定します
- `<link rel="icon" type="image/svg+xml" href="./favicon.svg">`：SVG形式のfaviconを設定しています

**注意点**：`og:image`に指定している`ogp-image.svg`はSVG形式のプレースホルダーです。多くの主要なSNS・チャットサービスはOGP画像としてSVG形式を正しく表示できない場合があります。実際の公開前には、`images/ogp-image.svg`をベースにデザインした1200×630pxのPNGまたはJPG画像を用意し、`og:image`の参照先を`./images/ogp-image.png`のように変更することを強く推奨します。

また、`canonical`タグや`og:url`は現時点では相対パス表記になっています。実際にドメインを取得して公開する際は、`https://www.grow-welfare.example.com/index.html`のような絶対URLに書き換える必要があります。

## 11. Firebase Hostingへのデプロイ手順（Windows 11想定）

すでにFirebase CLIのセットアップと`firebase init`が完了しており、`firebase.json`が以下のような内容になっていることを前提とします。

```json
{
  "hosting": {
    "public": "public",
    "ignore": [
      "firebase.json",
      "**/.*",
      "**/node_modules/**"
    ]
  }
}
```

この場合、`public`フォルダの中身がそのままFirebase Hosting上に公開されます。以下の手順で、本サイトのファイルを反映させてください。

**手順1：publicフォルダの中身を空にする**

Firebase初期化時に自動生成されたデフォルトのウェルカムページ（`index.html`など）が`public`フォルダに入っているため、これらを削除します。PowerShellでプロジェクトのルートフォルダに移動し、以下を実行します。

```powershell
Remove-Item public\* -Recurse -Force
```

**手順2：本サイトのファイルをpublicフォルダにコピーする**

ダウンロードした`grow-fukushi`フォルダ（ZIPを解凍したもの）の中身一式を、`public`フォルダにコピーします。例えば`grow-fukushi`フォルダがダウンロードフォルダにある場合は以下のように実行します。

```powershell
Copy-Item "$env:USERPROFILE\Downloads\grow-fukushi\*" -Destination "public" -Recurse -Force
```

コマンドプロンプトを使用する場合は以下のようになります。

```cmd
xcopy "%USERPROFILE%\Downloads\grow-fukushi\*" "public\" /E /Y
```

**手順3：publicフォルダの中身を確認する**

```powershell
dir public
```

以下のようなファイル・フォルダが表示されていれば正しくコピーされています。

```
about.html
contact.html
favicon.svg
images
index.html
recruit.html
script.js
services.html
style.css
```

**手順4：デプロイを実行する**

```powershell
firebase deploy --only hosting
```

コマンド実行後、デプロイが完了すると、`Hosting URL:` として `https://（プロジェクトID）.web.app` のようなURLが表示されます。

**手順5：公開されたサイトを確認する**

表示されたURLにブラウザでアクセスし、トップページ・法人案内・事業紹介・採用情報・お問い合わせの各ページが正しく表示され、リンクやナビゲーション、フォームの動作に問題がないかを確認してください。

## 12. GitHubリポジトリとの連携・自動デプロイについて

Firebase CLIの`firebase init hosting:github`コマンドを使用してGitHub Actionsによる自動デプロイを設定した場合、リポジトリのブランチにプッシュ（`git push`）するたびに、自動的にビルド・デプロイが実行される仕組みが構築されます。一般的には以下のような構成になります。

## 13. 画像・プレースホルダーの差し替え方法

本サイトで使用している画像は、著作権的に安全な自作のSVGイラストのみで構成されており、すべて`images`フォルダおよびルート直下の`favicon.svg`にまとめられています。実際の運用にあたっては、以下の画像を実際の写真やロゴに差し替えることを推奨します。

| ファイル | 用途 | 差し替えの推奨形式 |
|---|---|---|
| `favicon.svg` | ブラウザタブに表示されるアイコン | 法人の正式ロゴがある場合はSVGまたはPNG（32×32、180×180など複数サイズ用意すると尚良い） |
| `images/ogp-image.svg` | SNSシェア時に表示される画像 | 1200×630pxのPNGまたはJPG |

なお、代表理事の山﨑亮汰氏をはじめとする実在の人物の写真を掲載する際は、本人からの肖像権に関する使用許諾を得た上で掲載してください。

## 14. お問い合わせフォームの仕組みと注意点（バックエンド未実装について）

**重要な注意事項**：現時点の`contact.html`および`script.js`に実装されているお問い合わせフォームは、**フロントエンド（ブラウザ側）における入力チェックのみ**を行うものであり、実際にサーバーへメールを送信したり、データベースへ保存したりする処理は一切含まれていません。

具体的には、フォーム送信ボタンが押されると、`script.js`内の`initContactForm()`関数が以下の処理を行います。

1. 氏名・フリガナ・メールアドレス・お問い合わせ種別・お問い合わせ内容が空欄でないかをチェックする。
2. メールアドレスが正規表現パターン（`/^[^\s@]+@[^\s@]+\.[^\s@]+$/`）に一致する形式かどうかをチェックする。
3. 電話番号は任意項目のため、未入力であればチェックをスキップし、入力されていれば簡易的な桁数・文字種チェックを行う。
4. すべてのチェックを通過した場合は、あたかも送信が完了したかのような成功メッセージを画面に表示し、フォームをリセットする。

つまり、現状ではフォームに入力して送信ボタンを押しても、**法人の担当者に実際にメールは届きません。** 実運用でこのフォームを使用する場合は、以下のいずれかの方法でバックエンド処理を追加する必要があります。

- **Firebase Functions（Cloud Functions for Firebase）を利用する方法**：フォームの送信処理を`fetch`でCloud Functionsのエンドポイントに送信し、Functions側でNodemailerなどを用いてメール送信を行う。Firebase Hostingと同じプロジェクト内で完結できるため親和性が高い。
- **外部フォーム送信サービス（例：Googleフォーム連携、フォームメーラー系のSaaS）を利用する方法**：`<form>`タグの`action`属性に外部サービスのエンドポイントURLを指定し、そちらでメール転送処理を行う。
- **独自のサーバーサイドAPIを構築する方法**：PHPやNode.js（Express）などでメール送信APIを作成し、`fetch`で送信する。

いずれの方法を採用する場合も、現在実装されているフロントエンドの入力チェック機能（`initContactForm()`関数）はそのまま活用できます。送信処理の部分（現在は成功メッセージを表示するだけの箇所）に、実際の送信APIを呼び出す`fetch`処理を追加する形で拡張することを想定した設計になっています。

また、個人情報を含むお問い合わせ内容を実際に送受信する場合は、通信の暗号化（HTTPS。Firebase Hostingは標準でHTTPS対応です）に加え、個人情報保護方針に基づいた適切なデータの取り扱い・保管・破棄のルールを法人内で整備することを推奨します。

## 15. 事業紹介ページの「構想中」表記に関する重要な注意

福祉サービスは、実際に指定を受けて開設していない事業を、あたかも現在利用可能であるかのように表示することが、利用希望者・その家族・関係機関に対して誤解や不利益を与える可能性があるため、表記には細心の注意が必要です。

本サイトの`services.html`および`index.html`では、以下のルールで表記を統一しています。

- **開設中（`status-active`クラス、緑系のバッジ）**：現在実際にサービス提供を行っている事業。現時点では「放課後等デイサービス『みらい』」のみがこれに該当します。
- **構想中（`status-planning`クラス、オレンジ系のバッジ）**：法人として将来的な開設を検討しているが、指定申請や具体的な開設準備には至っていない事業。現時点では16事業がこれに該当します。

今後、法人の事業展開の進捗に応じて、以下のような表記変更が必要になった場合は、該当するHTML箇所を手動で編集してください。

- **構想中 → 開設準備中に変更する場合**：`services.html`内の該当する`.service-card`要素の`<span class="service-status status-planning">構想中</span>`を`<span class="service-status status-planning">開設準備中</span>`のようにテキストのみ変更するか、新たに`status-preparing`のようなクラスを`style.css`に追加してバッジの色分けを増やすことも可能です。
- **構想中・開設準備中 → 開設中に変更する場合**：該当する`.service-card`要素のクラスを`status-planning`から`status-active`に変更し、`data-status`属性も`planning`から`active`に変更してください。あわせて、テキストを「構想中」または「開設準備中」から「開設中」に変更してください。

この表記の正確性は、利用者保護・法令遵守の観点から非常に重要です。サイト更新時には、必ず法人内の事業担当者・管理者に最終確認を取った上で公開することを強く推奨します。

## 16. レスポンシブ対応の考え方

`style.css`のセクション7（レスポンシブ対応）では、以下の3段階のブレイクポイントを設定しています。

- **960px以下**：タブレット相当。ヒーローセクションが1カラムに変化し、代表理事メッセージのレイアウトも縦積みになります。カードグリッドは2カラムに変化します。
- **780px以下**：スマートフォン相当。この幅からハンバーガーメニューが表示され、グローバルナビゲーションは画面右側からスライドインするドロワーメニューに切り替わります。
- **640px以下**：小型スマートフォン相当。カードグリッドが1カラムに、フォームの2カラムレイアウトが1カラムに変化します。

レイアウトはCSS Gridおよびflexboxを中心に構築しており、特定のピクセル値に依存しすぎないよう、`clamp()`関数を用いた可変フォントサイズ（見出しなど）も一部採用しています。これにより、指定した2つの画面幅の間で、フォントサイズがなめらかに変化します。

## 18. よくあるトラブルと対処法（トラブルシューティング）

**Q. デプロイしたのに、Firebaseのデフォルトのウェルカムページが表示される。**

A. `public`フォルダの中身が空、またはデフォルトの`index.html`のままになっている可能性があります。本READMEの「11. Firebase Hostingへのデプロイ手順」を参照し、`grow-fukushi`フォルダの中身を正しく`public`フォルダにコピーしてから、再度`firebase deploy --only hosting`を実行してください。

**Q. 画像が表示されない。**

A. `images`フォルダごとコピーされているか確認してください。HTMLファイル内の画像パスは`./images/〇〇.svg`という相対パスで記述されているため、`images`フォルダが同じ階層に存在しない場合、画像は表示されません。

**Q. ハンバーガーメニューが開かない、または閉じない。**

A. `script.js`が正しく読み込まれているか、ブラウザの開発者ツール（F12キー）のコンソールにエラーが出ていないかを確認してください。ファイル名や拡張子（`.js`）を変更した場合は、各HTMLファイル末尾の`<script src="./script.js"></script>`の記述も合わせて修正が必要です。

**Q. お問い合わせフォームを送信しても、実際にメールが届かない。**

A. これは不具合ではなく仕様です。本READMEの「14. お問い合わせフォームの仕組みと注意点」に記載の通り、現状はフロントエンドの入力チェックのみを実装しており、実際の送信処理（バックエンド）は別途構築する必要があります。

**Q. スマートフォンで見ると、レイアウトが崩れる。**

A. 使用しているスマートフォンのブラウザが古いバージョンである可能性があります。可能であればブラウザを最新版にアップデートしてください。それでも崩れる場合は、該当箇所のCSS（`style.css`のセクション7）を確認し、必要に応じてブレイクポイントの調整を行ってください。

## 19. 今後の改善候補・拡張アイデア

現状のサイトはあくまで初期リリース版という位置づけであり、今後の運用の中で以下のような改善・拡張が考えられます。

- お問い合わせフォームのバックエンド実装（Firebase Functions等を用いたメール送信機能の追加）
- OGP画像のPNG/JPG化による、より確実なSNSシェア表示への対応
- 実際の写真素材への差し替え（代表理事プロフィール写真、施設内観・外観、活動風景など）
- お知らせ機能のCMS化（現状はHTMLへの直接記述による更新のため、更新担当者がHTML編集に不慣れな場合は、microCMSやContentful等のヘッドレスCMSとの連携を検討する余地があります）
- 多言語対応（英語表記の追加など、外国籍の利用者・保護者への情報提供の拡充）
- 採用情報ページへの応募フォームの独立化（現状はお問い合わせページに集約していますが、採用特化のフォーム項目を別途設けることも可能です）
- サイト内検索機能の追加
- Googleアナリティクス等のアクセス解析タグの導入（現状は導入していません。導入する場合はプライバシーポリシーへの明記も忘れずに行ってください）
- 構造化データ（JSON-LD）の追加によるSEOのさらなる強化（NPO法人・団体としてのSchema.org対応など）
